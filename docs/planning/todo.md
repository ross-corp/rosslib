- [ ] Add compound index on `threads(book, created DESC)`: create a new PocketBase migration in `api/migrations/` that adds a compound index on `(book, created DESC)` to the `threads` collection. The handler in `api/handlers/threads.go` queries `WHERE t.book = ? ORDER BY t.created DESC` but only has a single-column index on `book`, making the sort unindexed. Follow the existing migration pattern in `api/migrations/1700000020_add_performance_indexes.go`.
- [ ] Add compound index on `reports(status, created DESC)`: create a new PocketBase migration in `api/migrations/` that adds a compound index on `(status, created)` to the `reports` collection. The admin handler in `api/handlers/reports.go` filters by status and orders by `created DESC`, but the `reports` collection only has a single-column index on `status`.
- [ ] Include `blocks` records in account data deletion: in `api/handlers/userdata.go`, the `DeleteAllData` and `DeleteAccount` handlers delete many related records but skip the `blocks` collection. Add deletion of blocks where either `blocker = userId` or `blocked = userId` before the existing cleanup loop. This prevents orphaned block records when a user deletes their data. Follow the same pattern used for other collections in that handler.
- [ ] Include `saved_searches` and `reading_sessions` in account data deletion: in `api/handlers/userdata.go`, the `DeleteAllData` and `DeleteAccount` handlers are missing `saved_searches`, `reading_sessions`, `book_quotes`, and `reading_goals` from the list of collections to delete. Add these collection names to the deletion loop (they all have a `user` field that links to the user being deleted).
- [ ] Add `progress_percent` bounds validation to AddBook and UpdateBook: in `api/handlers/userbooks.go`, the `progress_percent` field accepted by `AddBook()` and `UpdateBook()` has no range validation. Add a check: if `progress_percent < 0 || progress_percent > 100`, return `400 { "error": "Progress percent must be between 0 and 100" }`. Similarly, reject negative `progress_pages` values.
- [ ] Clean up orphaned `genre_ratings` when deleting a user_book: in `api/handlers/userbooks.go`, the `DeleteBook()` handler cascades deletes to `collection_items` and `book_tag_values` but does not delete `genre_ratings` for the same user+book combination. After the existing cascade deletions, add a query to delete from `genre_ratings` where `user = userId AND book = bookId` to prevent orphaned records.
- [ ] Clean up orphaned `review_likes` when deleting a user_book: in `api/handlers/userbooks.go`, the `DeleteBook()` handler does not clean up `review_likes` referencing the deleted review. After the existing cascade deletions, add a query to delete from `review_likes` where `book = bookId AND review_user = userId` to prevent orphaned like records.
- [ ] Add index on `book_links(from_book, deleted_at)` for soft-delete queries: create a new PocketBase migration in `api/migrations/` that adds a compound index on `(from_book, deleted_at)` to the `book_links` collection. The handlers in `api/handlers/links.go` query `WHERE bl.from_book = ? AND (bl.deleted_at IS NULL OR bl.deleted_at = '')` but only have a unique index that doesn't cover the soft-delete filter. Follow the pattern in `api/migrations/1700000020_add_performance_indexes.go`.
- [ ] Add index on `review_comments(book, review_user, deleted_at)` for soft-delete queries: create a new PocketBase migration in `api/migrations/` that adds a compound index on `(book, review_user, deleted_at)` to the `review_comments` collection. The handler in `api/handlers/reviewcomments.go` queries with `WHERE rc.book = ? AND rc.review_user = ? AND (rc.deleted_at IS NULL OR rc.deleted_at = '')`. The existing index `idx_review_comments_review` on `(book, review_user)` doesn't cover the soft-delete predicate.
- [ ] Validate avatar/banner file type on upload: in `api/handlers/users.go`, the `UploadAvatar()` and `UploadBanner()` handlers accept any file from `e.Request.FormFile()` without checking the MIME type. Add validation to only accept `image/jpeg`, `image/png`, `image/gif`, and `image/webp` MIME types. Return `400 { "error": "Only JPEG, PNG, GIF, and WebP images are allowed" }` for other types.
- [ ] Clean up orphaned `book_quotes` when deleting a user_book: in `api/handlers/userbooks.go`, the `DeleteBook()` handler cascades deletes to `collection_items`, `book_tag_values`, `genre_ratings`, and `review_likes` but does not delete `book_quotes` for the same user+book combination. After the existing cascade deletions, add a query to delete from `book_quotes` where `user = userId AND book = bookId` to prevent orphaned quote records.
- [ ] Clean up orphaned `reading_sessions` when deleting a user_book: in `api/handlers/userbooks.go`, the `DeleteBook()` handler does not clean up `reading_sessions` referencing the deleted book. After the existing cascade deletions, add a query to delete from `reading_sessions` where `user = userId AND book = bookId` to prevent orphaned session records.
- [ ] Clean up orphaned `review_comments` when deleting a user_book: in `api/handlers/userbooks.go`, the `DeleteBook()` handler does not clean up `review_comments` where other users have commented on the deleted review. After the existing cascade deletions, add a query to delete from `review_comments` where `book = bookId AND review_user = userId` to prevent orphaned comment records.
- [ ] Add `rating` bounds validation to AddBook and UpdateBook: in `api/handlers/userbooks.go`, both `AddBook()` and `UpdateBook()` accept a `rating` field but don't validate its range. Add a check: if `rating` is provided and `rating < 1 || rating > 5`, return `400 { "error": "Rating must be between 1 and 5" }`. Currently, any integer is accepted, which could corrupt data displayed on profile and book pages.
- [ ] Add `new_follower` notification preference field: in `api/migrations/`, add a `new_follower` boolean field (default true) to the `notification_preferences` collection. This will be needed by the "Send notification when someone follows you" task, but the migration should be done first as a prerequisite. Follow the pattern used for existing fields like `review_liked` and `thread_mention` in the migration that creates `notification_preferences`.
- [ ] Batch shelf item counts into a single query in GetMyShelves: in `api/handlers/collections.go`, the `GetMyShelves()` function runs a separate `SELECT COUNT(*)` query for each shelf to get its item count (N+1 pattern). Replace the per-shelf count loop with a single query: `SELECT collection_id, COUNT(*) FROM collection_items GROUP BY collection_id WHERE collection_id IN (?)`, then map the counts back to shelves. This reduces database round-trips from N+1 to 2 for users with many shelves.
- [ ] Batch tag value loading into a single query in GetTagKeys: in `api/handlers/tags.go`, the `GetTagKeys()` function runs a separate query for each tag key to load its values (N+1 pattern). Replace the per-key value loop with a single query: `SELECT * FROM tag_values WHERE tag_key_id IN (?) ORDER BY tag_key_id, name`, then group the results by `tag_key_id` in Go. This reduces database round-trips from N+1 to 2 for users with many label categories.
- [ ] Batch vote counts and user-voted status into the book links query: in `api/handlers/links.go`, the `GetBookLinks()` function runs separate `SELECT COUNT(*)` and `SELECT id` queries for each link to get vote counts and the viewer's vote status (N+1 pattern). Rewrite to use a single query with `LEFT JOIN (SELECT book_link_id, COUNT(*) AS votes FROM book_link_votes GROUP BY book_link_id)` and a second `LEFT JOIN` for the viewer's vote, eliminating the per-link loops.
- [ ] Include `review_likes`, `review_comments`, `recommendations`, `computed_collections`, `feedback`, `reports`, and `api_tokens` in account data deletion: in `api/handlers/userdata.go`, both `DeleteAccount` (line 23-42) and `DeleteAllData` (line 95-113) are missing these collections from the deletion loop. Add them to the `userOwned` slice. For `recommendations`, also delete where `recipient = userId` (not just `user`). For `reports`, also delete where `reviewer = userId`. Note: `DeleteAllData` is also missing `recommendations` entirely (DeleteAccount has it but DeleteAllData does not). This is separate from the existing todo item for `blocks`/`saved_searches`/`reading_sessions`.
- [ ] Add `note` max length validation to CreateBookLink: in `api/handlers/links.go`, the `CreateBookLink()` handler accepts a `note` field but never validates its length. Add a check: if `len(note) > 2000`, return `400 { "error": "Note must be 2,000 characters or fewer" }`. This matches the max length pattern used for review comment body and quote text elsewhere in the codebase.
- [ ] Add `date_read` and `date_started` format validation to UpdateBook: in `api/handlers/userbooks.go`, the `UpdateBook()` handler (around line 153-160) accepts `date_read`, `date_dnf`, and `date_started` as string fields but sets them on the record without validating that they're valid dates. Add `time.Parse(time.RFC3339, dateStr)` validation for each date field when provided, returning `400 { "error": "Invalid date format for <field>" }` on failure. The `CreateSession` handler in `api/handlers/sessions.go` already validates dates this way — apply the same pattern to UpdateBook.
